How to create a new skill

A skill is a slash command that you invoke in Claude Code (e.g., /my-skill). Each skill lives in its own directory under ~/.claude/skills/.

1Create the skill directory

BASH 
mkdir ~/.claude/skills/my-skill

2Create SKILL.md with frontmatter

MARKDOWN 
---
name: my-skill
description: Short description of what this skill does
argument-hint: "<target> [--verbose]"
tags: [category1, category2]
user-invocable: true
---

# My Skill

Instructions for Claude Code to follow when this skill is invoked.

## What To Do
1. First step
2. Second step

## Arguments
- `<target>`: What to operate on

3Test your skill

CLAUDE 
/my-skill some-argument

Frontmatter field reference

FieldRequiredDescription
nameYesThe slash command name (without the /)
descriptionYesOne-line description shown in skill listings
argument-hintNoHint text shown after the command name
tagsNoArray of tags for categorization
agentNoName of an agent to activate when this skill runs
user-invocableNoSet to true to make the skill available as a slash command

Real example: the /tdd skill

MARKDOWN 
---
name: tdd
description: Start TDD workflow for a feature
agent: tdd-guide
user-invocable: true
---

Implement the following using strict TDD (RED-GREEN-REFACTOR): $ARGUMENTS.
Start with interface definition, then write failing tests, then minimal implementation, then refactor.

The $ARGUMENTS placeholder is replaced with whatever the user types after /tdd. The agent: tdd-guide field activates the TDD agent from ~/.claude/agents/tdd-guide.md.

How to create a new agent

An agent is a Markdown file that defines specialized behavior for Claude Code. Agents live in ~/.claude/agents/.

1Create the agent file

BASH 
touch ~/.claude/agents/my-agent.md

2Write agent definition with frontmatter

MARKDOWN 
---
name: my-specialist
description: Expert in [domain] for [specific tasks]
tools: Read, Write, Edit, Bash, Grep, Glob
model: claude-sonnet-4-6
mode: subagent
---

You are a specialist in [domain]. You are called as a subagent by the orchestrator
when tasks require [specific expertise].

## Workflow
1. First, analyze the request
2. Then, implement the solution
3. Finally, verify the result

## Rules
- Only read and edit files under [allowed scope]
- Never run destructive commands without confirmation
- Output a structured report at the end

3Reference the agent from a skill (optional)

YAML 
agent: my-specialist

Frontmatter field reference

FieldRequiredValuesDescription
nameYesstringIdentifier used when referencing this agent
descriptionYesstringOne-line summary. The orchestrator reads this for routing.
toolsYescomma-separatedAllowed tools: Read, Write, Edit, Bash, Grep, Glob, Task, etc.
modelYessee belowClaude model for this agent
modeYesprimary / subagent / secondaryExecution context

Model values

claude-opus-4-6Complex reasoning, architecture
claude-sonnet-4-6Balanced -- most agents use this
claude-haiku-4-5Fast, token-efficient tasks

How to add a hook

Hooks intercept Claude Code tool calls at specific lifecycle points. A hook exits 0 to allow, 2 to block, or prints warnings on stdout.

Hook event reference

EventWhen it firesCan block?Use for
PreToolUseBefore any tool call executesYes (exit 2)Blocking dangerous writes, enforcing scan-first policies
PostToolUseAfter a tool call completesNoPrinting reminders, logging, coverage nudges
StopWhen the session endsNoPersisting session state
SessionStartWhen a new session opensNoLoading context summaries
PreCompactBefore context compactionNoSaving work-in-progress notes

1Write the hook script

PYTHON 
# ~/.claude/hooks/my-hook.py
import json, sys

payload = json.load(sys.stdin)
tool_name = payload.get("tool_name", "")
tool_input = payload.get("tool_input", {})

# Example: block Write calls to .env files
if tool_name == "Write":
    file_path = tool_input.get("file_path", "")
    if file_path.endswith(".env"):
        print("Blocked: do not write secrets to .env files.", file=sys.stderr)
        sys.exit(2)   # exit 2 = block the tool call

sys.exit(0)  # exit 0 = allow

2Register the hook in settings.json

JSON 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          { "type": "command", "command": "python3 ~/.claude/hooks/my-hook.py" }
        ]
      }
    ]
  }
}

3Verify the hook fires

BASH 
# Open a new Claude Code session, then attempt the blocked action:
# Ask Claude to write a file ending in .env -- the hook must block it.

How to add an MCP server

MCP servers extend Claude Code with external tools that communicate via stdio. Configure them in ~/.claude/.mcp.json.

1Open ~/.claude/.mcp.json

BASH 
code ~/.claude/.mcp.json

2Add the server entry

JSON 
{
  "mcpServers": {
    "my-server": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-my-server"],
      "env": {
        "MY_API_KEY": "${MY_API_KEY}"
      }
    }
  }
}

3Set environment variables

BASH 
# Add to ~/.bashrc or ~/.zshrc (never hardcode in .mcp.json)
export MY_API_KEY="your-key-here"

4Add tool permissions in settings.json

JSON 
"permissions": {
  "allow": [
    "mcp__my-server__tool_name"
  ]
}

5Restart Claude Code and verify

CLAUDE 
/mcp
# Lists all connected MCP servers and their tools

How to run the DSL engine

The DSL engine parses shortcut commands like @arch:ms @deploy:k8s --scale=3 into structured execution plans. It supports 179 skills across 46 categories.

1Install dependencies

BASH 
cd C:/.claude/engine
npm install && npm run build

2Run a DSL command

BASH 
npx ts-node src/index.ts "@arch:ms @deploy:k8s --scale=3"

3Combine multiple categories

BASH 
# Full microservice with tests and K8s deployment:
npx ts-node src/index.ts "@arch:ms @dotnet @test:unit @test:int @deploy:k8s --scale=3"

# Code review and debugging:
npx ts-node src/index.ts "@dotnet @debug:perf @api:rest --target=src/backend/"

Available DSL categories
